## 추가 리팩토링 티켓 — 스캔 품질 & 공격 안전성

---

### 관점 A — "실제로 스캔이 되고 있나? 로그만 찍히는 건 아닌가?"

---

**[SCAN-01] `xss_scan.py` — Stored XSS 탐지 코드 전체가 데드코드**

- **파일**: `s2n/s2nscanner/plugins/xss/xss_scan.py`
- **문제**: `_test_stored()`, `_record_stored()` 메서드가 정의되어 있지만 실행 흐름 어디에서도 호출되지 않습니다. `run()` → `_scan_single_url()` → `_scan_input_point()` → `_scan_parameter()` → `_test_payload_on_param()` 체인에서 `_test_stored()`는 완전히 빠져있습니다. Stored XSS는 선언만 있고 실제로는 전혀 탐지되지 않는 상태입니다.
- **영향**: 사용자는 XSS 플러그인을 실행했다고 생각하지만 Stored XSS는 완전히 누락됩니다. README나 문서에 Stored XSS 탐지가 포함되어 있다면 허위 기능이 됩니다.

---

**[SCAN-02] `csrf_scan.py` — 토큰 키워드 유무만으로 취약점 판정 (서버 검증 여부 미확인)**

- **파일**: `s2n/s2nscanner/plugins/csrf/csrf_scan.py`
- **문제 1 (False Positive)**: HTML에 "csrf", "token" 문자열이 없으면 취약하다고 판정합니다. 그러나 서버가 `SameSite=Strict` 쿠키나 `X-Requested-With` 헤더로 CSRF를 방어하는 경우 토큰 키워드가 없어도 실제로는 안전합니다.
- **문제 2 (False Negative)**: 반대로 `<input type="hidden" name="csrf_token" value="ABC">` 처럼 토큰이 HTML에 있더라도 서버가 검증을 하지 않는다면 실제로는 취약합니다. 현재 코드는 서버가 실제로 토큰을 검증하는지 전혀 확인하지 않습니다.
- **문제 3 (중복 탐지)**: `scan_html()`과 `scan_form_tags()` 두 함수가 거의 동일한 키워드를 거의 동일한 HTML에서 각각 탐색합니다. 하나가 "안전", 다른 하나가 "취약"으로 나오는 혼란스러운 결과가 가능합니다.
- **핵심**: 이 플러그인은 스캔이라기보다 정적 HTML 키워드 검사에 가깝습니다.

---

**[SCAN-03] `soft_brute_force_main.py` — 요청이 전부 실패해도 취약점 Finding 생성 (False Positive)**

- **파일**: `s2n/s2nscanner/plugins/soft_brute_force/soft_brute_force_main.py`
- **문제**: `responses = []` 인 경우 (네트워크 오류 등으로 모든 POST 요청 실패) 응답이 하나도 없음에도 429도 없고 차단 키워드도 없다는 이유로 "No Rate Limiting Detected" — 즉 취약점 발견이라는 Finding을 생성합니다. 실제로는 요청 자체가 나가지 않은 상황인데 취약하다고 리포팅하는 겁니다.
- **수정**: 요청이 하나도 성공하지 않았을 경우(`not responses`)를 별도 분기로 처리해 SKIPPED 또는 에러로 처리해야 합니다.

---

**[SCAN-04] `soft_brute_force_main.py` — 로그인 폼 필드명 하드코딩으로 실제 인증 시도가 안 될 수 있음**

- **파일**: `s2n/s2nscanner/plugins/soft_brute_force/soft_brute_force_main.py`
- **문제**: `data = {"username": f"invalid_user_{i}", "password": f"invalid_pass_{i}"}` 로 필드명이 고정되어 있습니다. 실제 대상 서버가 `user`, `email`, `login`, `passwd` 등 다른 필드명을 사용한다면 이 요청은 로그인 시도로 인식조차 되지 않습니다. Rate Limiting을 올바르게 트리거하지 못하므로 결과 자체가 신뢰 불가합니다.
- **수정**: `soft_brute_force`도 `file_upload_utils.py`처럼 폼 파싱을 거쳐 실제 필드명을 추출해야 합니다.

---

**[SCAN-05] `sqli_scan.py` — Time-Based Blind에서 네트워크 레이턴시와 SLEEP 구분 불가 + 에러 묵살**

- **파일**: `s2n/s2nscanner/plugins/sqlinjection/sqli_scan.py`
- **문제 1**: `if elapsed_time > TIME_THRESHOLD` 단순 비교로 취약점을 판정합니다. 서버가 느리거나 네트워크가 불안정한 환경에서 SLEEP 없이도 TIME_THRESHOLD를 초과하면 False Positive가 발생합니다. 기준이 되는 베이스라인 응답 시간을 먼저 측정해야 합니다.
- **문제 2**: `except Exception as e: pass` 로 예외를 완전히 무시합니다. 요청 타임아웃(즉 실제로 10초를 기다린 것)과 다른 네트워크 오류를 동일하게 처리해 정확한 판정이 불가능합니다.

---

**[SCAN-06] `brute_force_main.py` — `requests_sent` 항상 0 (실제 요청 수 미반영)**

- **파일**: `s2n/s2nscanner/plugins/brute_force/brute_force_main.py`
- **문제**: `requests_sent = 0` 으로 초기화 후 Selenium으로 실제 로그인을 시도하는 `scan_brute_force_with_selenium()`을 호출하지만, 이 함수가 반환하는 값을 `requests_sent`에 누적하지 않습니다. 최종 `PluginResult`에는 `requests_sent=0`이 담깁니다. ScanReport 전체의 요청 수 통계를 신뢰할 수 없게 만듭니다.

---

**[SCAN-07] `oscommand_main.py` — Windows 환경 페이로드 부재 (탐지 범위 불완전)**

- **파일**: `s2n/s2nscanner/plugins/oscommand/oscommand_main.py`
- **문제**: `DEFAULT_PAYLOADS`가 전부 Unix/Linux 기반 명령어 (`;id`, `|whoami`, `|uname -a`, `;echo vulnerable`)이고, `DEFAULT_PATTERNS` 역시 `uid=\d+`, `root:.*:0:0:` 등 Linux 출력 패턴만 있습니다. Windows 서버(`& whoami`, `& dir`, `& ipconfig /all`)는 탐지가 아예 불가능합니다.

### 관점 B — "실제 공격이 일어나고 있나? (일어나면 안 되는데)"

---

**[SAFETY-01] `file_upload_utils.py` — 업로드된 테스트 파일 서버 측 미정리 (웹셸 잔류)**

- **파일**: `s2n/s2nscanner/plugins/file_upload/file_upload_utils.py`
- **문제**: `finally` 블록에서 로컬 임시 파일(`os.remove(test_path)`)만 삭제합니다. 서버에 실제로 업로드된 `.php`, `.jpg.php` 파일은 스캔 종료 후에도 서버에 그대로 남아 있습니다. 스캐너가 취약점을 확인하기 위해 올려놓은 파일이 실제 웹셸처럼 잔류하게 됩니다. 스캔 후 식별된 URL로 DELETE 요청을 시도하거나, 업로드 성공 파일 목록을 리포트에 명시해 수동 정리를 안내해야 합니다.

---

**[SAFETY-02] `soft_brute_force_main.py` — 계정 잠금 트리거 후 복구 로직 없음**

- **파일**: `s2n/s2nscanner/plugins/soft_brute_force/soft_brute_force_main.py`
- **문제**: Rate Limiting이 없다는 것을 확인하기 위해 실제로 10번 연속 빠른 로그인 실패를 서버에 보냅니다. 서버에 Account Lockout 정책이 있다면 이 10번의 시도로 실제 계정이 잠길 수 있습니다. 잠금 발생 여부를 감지하더라도 잠금 해제 시도나 경고 없이 그냥 진행합니다. Rate Limiting 탐지와 Account Lockout 탐지를 구분하고, 잠금 발생 시 명시적인 경고를 주어야 합니다.

---

**[SAFETY-03] `brute_force_main.py` — 성공/실패와 관계없이 Selenium 세션 미정리 (서버 측 좀비 세션)**

- **파일**: `s2n/s2nscanner/plugins/brute_force/brute_force_main.py`
- **문제**: `finally: driver.quit()` 으로 브라우저는 닫지만, 서버에 생성된 로그인 세션(쿠키)은 명시적으로 로그아웃하지 않습니다. 20번의 로그인 시도 중 성공한 세션이 서버에 남아 있게 됩니다. 또한 `requests_sent = 0`이지만([SCAN-06]과 연계) 실제로는 최소 20회 이상의 HTTP 요청이 대상 서버에 전달됩니다. 리포트에는 0건으로 기록되지만 서버 로그에는 대량의 로그인 시도가 남습니다.

---

**[SAFETY-04] `xss_scan.py` — Stored XSS 코드 활성화 시 서버 측 페이로드 미정리 설계**

- **파일**: `s2n/s2nscanner/plugins/xss/xss_scan.py`
- **문제**: [SCAN-01]에서 밝혔듯 `_test_stored()`는 현재 데드코드입니다. 그러나 이 코드를 활성화하면 `<script>alert('{unique_tag}')</script>`를 서버 DB에 저장한 뒤 정리하는 로직이 전혀 없습니다. Stored XSS 페이로드가 DB에 남으면 이후 해당 페이지를 방문하는 모든 사용자(테스터 포함)에게 스크립트가 실행됩니다. SCAN-01에서 이 코드를 활성화하기 전에 반드시 페이로드 정리 로직(삽입 후 DELETE 또는 업데이트)을 먼저 설계해야 합니다.

---

**[SAFETY-05] `oscommand_main.py` — 커스텀 페이로드 안전성 검증 없음**

- **파일**: `s2n/s2nscanner/plugins/oscommand/oscommand_main.py`
- **문제**: `self.payloads = self.config.get("payloads", DEFAULT_PAYLOADS)` 로 외부 config에서 임의의 페이로드를 주입받을 수 있습니다. 기본 페이로드는 `;id`, `|whoami` 같은 read-only 명령이지만, 사용자가 config에 `;rm -rf /tmp/*`, `| shutdown -h now` 같은 파괴적인 페이로드를 넣으면 스캐너가 그대로 실행합니다. 위험 커맨드 차단 리스트나 페이로드 안전성 검증 레이어가 전혀 없습니다.

### 관점 C — "플러그인끼리 인터페이스가 통일되어 있나?"

---

**[IFACE-01] `__init__` 파라미터 타입 불일치 — 7개 플러그인이 각자 다른 config 타입 사용**

- **파일**: `s2n/s2nscanner/plugins/*/XXX_main.py` (7개)
- **문제**: 플러그인마다 `__init__`의 config 파라미터 타입이 다릅니다.
  - `SQLInjectionPlugin`: `Optional[Dict[str, Any]]`
  - `XSSScanner`: `PluginConfig | None`
  - `CSRFScanner`: `Optional[PluginConfig]`
  - `BruteForcePlugin`: `Any`
  - `FileUploadPlugin`: `Optional[PluginContext]` (잘못된 타입)
  - `OSCommandPlugin`: `Optional[Dict[str, object]]`
  - `SoftBruteForcePlugin`: `Optional[Dict[str, Any]]`
- **영향**: `scan_engine.py`에서 플러그인을 동적으로 로드해 초기화할 때 타입 불일치로 런타임 오류가 발생할 수 있습니다. 특히 `FileUploadPlugin`은 `PluginContext`를 config로 받는 설계 오류가 있습니다.
- **수정**: 모든 플러그인의 config 파라미터 타입을 `Optional[PluginConfig]`으로 통일

---

**[IFACE-02] 헬퍼 함수 사용 불일치 — 일부 플러그인만 `helper.py` 활용**

- **파일**: `s2n/s2nscanner/plugins/helper.py`, 각 플러그인
- **문제**: `helper.py`에 `resolve_client`, `resolve_depth`, `resolve_target_url`이 공통 함수로 정의되어 있지만 사용 방식이 제각각입니다.
  - `sqlinjection`: `helper.py` 공통 함수 사용
  - `xss`, `csrf`, `file_upload`: `plugin_context.scan_context.http_client`에 직접 접근
  - `oscommand`, `soft_brute_force`: 동일 로직을 인스턴스 메서드로 중복 구현(`_resolve_*`)
- **영향**: 공통 함수의 버그 수정이나 시그니처 변경 시 직접 접근하는 플러그인들에 반영되지 않아 동작 불일치가 생깁니다.
- **수정**: 모든 플러그인이 `helper.py` 공통 함수를 사용하도록 통일

---

**[IFACE-03] 클래스명 및 `__init__.py` export 패턴 불일치**

- **파일**: `s2n/s2nscanner/plugins/*/__init__.py` (7개)
- **문제**: 플러그인별 클래스명과 export 방식이 통일되어 있지 않습니다.
  - `sqlinjection`: 클래스명 `SQLInjectionPlugin`, export `main as Plugin`
  - `xss`: 클래스명 `XSSScanner`, export `main as Plugin`
  - `csrf`: 클래스명 `CSRFScanner`, export `main as Plugin`
  - `soft_brute_force`: `SoftBruteForcePlugin`과 `main as Plugin` 이중 노출 (불필요한 중복)
- **영향**: `scan_engine`이 플러그인을 동적 로드할 때 어떤 심볼을 참조해야 하는지 일관되지 않습니다.
- **수정**: 모든 `__init__.py`에서 `from .xxx_main import main as Plugin`만 노출하도록 통일

---

**[IFACE-04] `OSCommandPlugin` 레거시 메서드 잔재**

- **파일**: `s2n/s2nscanner/plugins/oscommand/oscommand_main.py`
- **문제**: `OSCommandPlugin`에만 `initialize()`, `scan()`, `teardown()` 레거시 메서드가 남아 있습니다. 다른 6개 플러그인에는 없는 메서드로, 현재 `scan_engine.py`의 실행 흐름에서 호출되지 않습니다.
- **영향**: 코드를 읽는 사람이 이 메서드들이 실제로 사용되는지 혼동하게 됩니다. 인터페이스 계약을 명시적으로 정의할 때 이 메서드들이 충돌을 일으킬 수 있습니다.
- **수정**: `initialize()`, `scan()`, `teardown()` 제거 후 표준 `run()` 인터페이스로 통일

---

**[IFACE-05] 플러그인 공통 추상 베이스 클래스 부재**

- **파일**: 새 파일 `s2n/s2nscanner/plugins/base.py` + 각 플러그인
- **문제**: 플러그인 계약(어떤 메서드를 반드시 구현해야 하는지)이 코드 레벨에서 강제되지 않습니다. `scan_engine.py`가 `hasattr(plugin, "run")`, `hasattr(plugin, "pre_scan")` 같은 방어적 체크를 매번 수행해야 하는 이유입니다.
- **수정**: 아래와 같이 `BasePlugin` Protocol 또는 추상 클래스를 정의하고 각 플러그인이 이를 구현하도록 강제

```python
class BasePlugin(Protocol):
    name: str
    description: str

    def __init__(self, config: Optional[PluginConfig] = None) -> None: ...
    def run(self, plugin_context: PluginContext) -> PluginResult: ...
```
